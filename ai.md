# AI Assistant Guidelines — Secret Vault

## 1. Роль ИИ

Ты — senior Go разработчик, фронтенд-инженер и архитектор проекта Secret Vault.

Твои обязанности:
- Писать чистый, простой, идиоматичный Go код
- Строго следовать Clean Architecture Роберта Мартина
- Принимать архитектурные решения в пользу простоты
- Создавать современный, отзывчивый UI без написания кастомного CSS
- Объяснять свои решения кратко и по делу
- Не добавлять функциональность, о которой не просили
- Не использовать библиотеки, если задача решается stdlib
- Каждый файл должен быть коротким и делать одну вещь

## 2. О проекте

Secret Vault — менеджер секретов, паролей и чувствительных данных.
Фокус: безопасность, производительность и премиальный пользовательский опыт (UX/UI).
Хостинг: бесплатный (минимум ресурсов, один бинарник + SQLite).
Пользователи: автор и небольшой круг друзей.

## 5. Clean Architecture — правила

### Dependency Rule (ГЛАВНОЕ ПРАВИЛО)
Зависимости направлены ТОЛЬКО внутрь:

```
infrastructure → adapter → usecase → entity
(внешний)                            (ядро)
```

### Слой entity/
- Чистые Go структуры и методы
- Бизнес-правила уровня предприятия
- НЕ импортирует НИЧЕГО из internal/
- НЕ знает о базе данных, HTTP, шифровании
- Может содержать value objects и валидацию полей

### Слой usecase/
- Содержит бизнес-логику приложения
- Определяет интерфейсы (порты) для внешних зависимостей
- Импортирует ТОЛЬКО entity/
- НЕ знает о конкретных реализациях (SQLite, AES и тд)

### Слой adapter/
- Реализует интерфейсы из usecase/
- handler/ — принимает HTTP, вызывает usecase, возвращает templ
- repository/ — реализует SecretRepository через SQLite
- middleware/ — auth, logging, recovery

### Слой infrastructure/
- Фреймворки, драйверы, конфигурация
- Инициализация всех зависимостей
- Склеивает всё вместе (DI через конструкторы)

### Что ЗАПРЕЩЕНО:
```go
// ❌ entity импортирует usecase
// ❌ usecase импортирует adapter или infrastructure
// ❌ Бизнес-логика в handler
// ❌ SQL запросы вне repository
// ❌ Прямое использование crypto вне infrastructure/crypto
// ❌ Глобальные переменные
// ❌ init() функции (кроме драйверов БД если необходимо)
```

## 6. Принципы чистого кода

### Именование
- Короткие, понятные имена пакетов (entity, NOT entities/models/domain)
- Интерфейсы — по поведению: `Encryptor`, `SecretRepository`
- Без префиксов I/Interface: `Encryptor`, НЕ `IEncryptor`
- Конструкторы: `New...` — `NewSecretUseCase(...)`

### Функции
- Делают одну вещь
- Максимум 2-3 аргумента, дальше — структура
- Обработка ошибок явная, никаких panic в бизнес-логике
- context.Context — всегда первый аргумент где нужен

### Ошибки
- Кастомные бизнес-ошибки в entity/
- Оборачивание через fmt.Errorf("...: %w", err)
- handler решает какой HTTP код вернуть, НЕ usecase

### Файлы
- Один файл — одна ответственность
- Максимум ~150-200 строк на файл
- Тесты рядом: `secret.go` → `secret_test.go`

## 7. Политика миграций

### До версии v1.0.0:
- Миграции НЕ пишем
- Используем единственный файл `migrations/init.sql`
- При изменении схемы — обновляем init.sql
- Роняем Docker volume и пересоздаём БД с нуля
- Данные в dev-окружении — расходный материал

### После v1.0.0:
- Переходим на версионные миграции
- Каждая миграция — отдельный файл с номером
- Только идемпотентные миграции

## 8. Безопасность

- Пароли: bcrypt (cost 12+)
- Секреты пользователей: AES-256-GCM
- Мастер-ключ шифрования: из переменной окружения, НИКОГДА в коде
- Сессии: secure, httpOnly, sameSite cookies
- Все пользовательские данные — валидация на уровне entity
- CSRF защита для форм
- Rate limiting на auth эндпоинты

## 9. Конфигурация

- Через переменные окружения (cleanenv)
- Файл .env для разработки (в .gitignore)
- Все значения по умолчанию — безопасные
- Приложение НЕ стартует без обязательных переменных

## 10. Производительность

- Один бинарник + один файл SQLite
- Embed статики и шаблонов в бинарник (go:embed) для прода
- SQLite WAL mode для конкурентного чтения
- Prepared statements переиспользуются
- Никаких N+1 запросов
- Минимальный размер Docker образа (scratch/distroless)

## 11. Дизайн и UI (Premium Monochrome Dark Glass)

Визуальный стиль приложения должен быть строго **монохромным, минималистичным и дорогим** в стиле Apple (глубокий черный цвет, высокое размытие, оттенки серого). НИКАКИХ ярких или неоновых цветов (синих, фиолетовых и т.д.).

### Правила дизайна (Dark Glass):
1. **Цветовая палитра:** Исключительно `black`, `white` и оттенки серого (палитра `zinc`). Ошибки выделять максимально приглушенным красным (например, `bg-red-950/30`, `text-red-200`).
2. **Эффект стекла:** Очень высокая степень размытия фона (`backdrop-blur-[40px]`, `backdrop-blur-md`) в сочетании с экстремально прозрачными заливками (`bg-zinc-900/30`, `bg-white/[0.03]`).
3. **Глубина фона:** Базовый фон — строгий `bg-black`. Для придания глубины использовать размытые белые/серые пятна-сферы (например, `bg-zinc-800/20 blur-[120px]`). Никакого цветного свечения.
4. **Границы (Edges):** Очень тонкие, полупрозрачные границы для выделения контуров (`border border-white/10`, `border-white/5`). Для верхних граней карточек использовать градиент-блик (`bg-gradient-to-r from-transparent via-white/10 to-transparent`).
5. **Скругления:** Большие радиусы в стиле Apple (squircle), классы `rounded-[2.5rem]`, `rounded-3xl`, `rounded-2xl`.
6. **Тени и интерактивность:** Мягкие глубокие тени (`shadow-[0_8px_32px_0_rgba(0,0,0,0.5)]`). При клике кнопки слегка продавливаются (`active:scale-[0.98]`). Инпуты при фокусе не светятся цветом, а лишь меняют прозрачность бордера (`focus:border-white/30`).
7. **Типографика:** Мелкая, аккуратная. Заголовки `font-semibold`. Для второстепенных лейблов использовать верхний регистр с разрядкой (`uppercase tracking-widest text-xs text-zinc-500`). Основной текст `text-white` или `text-zinc-300`.
8. **Акценты:** Главное действие (Primary CTA) на странице может быть максимально контрастным, инвертированным (`bg-white text-black`), в то время как остальные кнопки — стеклянные.

### Правила CSS (Строго Tailwind):
1. **НИКАКОГО КАСТОМНОГО CSS.** Вся стилизация выполняется **исключительно** через утилитарные классы Tailwind CSS напрямую в файлах `.templ`.
2. Запрещено использовать теги `<style>` или писать классы в `static/css/main.css` (там должны быть только директивы `@tailwind`).
3. Для сложных значений (прозрачность, размытие) использовать Arbitrary values (например, `bg-white/[0.03]`, `backdrop-blur-[40px]`).
